<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Supabase Auth | Untrusted client initiates auth flows; session tokens must be validated server-side |
| Browser → Supabase Database | Client-side queries run with user's JWT; RLS policies enforce data isolation |
| Anonymous → Linked User | Identity linking must verify OAuth token belongs to the linking user |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Spoofing | supabaseAuth.ts | mitigate | Use Supabase's built-in token validation; never trust client-side `is_anonymous` flag alone — verify via session JWT |
| T-05-02 | Tampering | game_saves table | mitigate | RLS policy: `auth.uid() = player_id` ensures players can only modify their own saves; JSONB schema validated at application layer |
| T-05-03 | Repudiation | Auth session | accept | Anonymous sessions have no identity to repudiate; linked sessions use Supabase's auditable auth logs |
| T-05-04 | Information Disclosure | global_leaderboards | mitigate | RLS: SELECT for all (intentionally public), but INSERT/UPDATE/DELETE restricted to `auth.uid() = player_id` |
| T-05-05 | Denial of Service | Supabase free tier | accept | 500MB DB, 50k MAU sufficient for v1; monitor usage, add rate limiting if needed |
| T-05-06 | Elevation of Privilege | linkIdentity | mitigate | Supabase's linkIdentity requires active session; cannot link to another user's OAuth account without their token |
</threat_model>
