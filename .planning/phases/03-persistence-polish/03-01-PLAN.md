Step 2 — Create src/lib/idbStorage.ts:
Create a Zustand-compatible storage adapter using idb-keyval. The adapter must implement the StorageValue interface that Zustand's persist middleware expects:
```typescript
import { get, set, del, clear } from 'idb-keyval';

export const idbStorage = {
  getItem: async (name: string) => {
    const value = await get(name);
    return value ? JSON.parse(value) : null;
  },
  setItem: async (name: string, value: unknown) => {
    await set(name, JSON.stringify(value));
  },
  removeItem: async (name: string) => {
    await del(name);
  },
};
```
The store name is 'ai-idle-hacker-game'. Use idb-keyval's default database (no custom DB name needed).

Step 6 — Add offline progress calculation in src/hooks/useGameLoop.ts:
In the handleVisibility function, when the tab becomes visible again (document.hidden is false), calculate offline progress:

```typescript
const handleVisibility = () => {
  if (document.hidden) {
    storeRef.current.getState().setLastUpdate(Date.now());
  } else {
    // Tab became visible — calculate offline progress
    const store = storeRef.current.getState();
    const now = Date.now();
    const elapsed = (now - store.lastUpdate) / 1000; // seconds

    if (elapsed > 5) { // Only calculate if away for more than 5 seconds
      const rate = store.getCreditRate();

      // 8-hour cap with diminishing returns after 2 hours
      // Formula: min(elapsed, 8h) × rate × (elapsed > 2h ? 0.5 : 1.0)
      const cappedSeconds = Math.min(elapsed, 8 * 3600); // 8h = 28800s
      const diminishingMultiplier = elapsed > 2 * 3600 ? 0.5 : 1.0; // 2h = 7200s
      const offlineCredits = Math.floor(cappedSeconds * rate * diminishingMultiplier);

      if (offlineCredits > 0) {
        store.updatePlayer({
          credits: store.player.credits + offlineCredits,
        });
        store.addNotification(
          `Welcome back! Earned ${offlineCredits.toLocaleString()} credits while away (${Math.floor(elapsed / 60)}min)`,
          'info'
        );
      }
    }
  }
};
```

Per CONTEXT.md decision D-02: The formula is `min(elapsed, 8h) × rate × (elapsed > 2h ? 0.5 : 1.0)`. The diminishing multiplier applies to the ENTIRE capped duration when elapsed exceeds 2 hours (not just the portion after 2h).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → IndexedDB | Client-side storage, user can inspect/modify via DevTools |
| beforeunload → save | Unreliable event — browser may kill before save completes |
| System clock → offline progress | User can manipulate system clock to inflate offline earnings |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | IndexedDB storage | accept | Client-side game data; no server trust boundary. User modifying their own save is expected in single-player idle games. |
| T-03-02 | Spoofing | beforeunload handler | accept | beforeunload is best-effort; browser may not fire it. Auto-save every 30s provides fallback. No security impact. |
| T-03-03 | Tampering | Offline progress calculation | mitigate | 8-hour cap limits maximum exploitation. Diminishing returns after 2h reduces incentive to abuse. Server-side validation in Phase 5 (Supabase) will add authoritative time source. |
| T-03-04 | Information Disclosure | IndexedDB save data | accept | Contains only game state (credits, level, equipment). No PII, no credentials, no sensitive data. |
</threat_model>
